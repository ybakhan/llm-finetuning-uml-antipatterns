═══ ANTIPATTERN INSTANCE RULE ═══════════════════════════════════════════════
One antipattern instance = exactly ONE (base use case, inclusion use case) pair.

  • When a base use case includes N qualifying inclusion use cases, that is N
    separate instances — one per (base, inclusion) pair.
  • Do NOT group multiple inclusion use cases into a single instance entry.

Example: "Book Flight" includes "Validate Payment Details", "Process Payment", and
"Generate Ticket Confirmation" (none associated with an actor). This is THREE instances:
  Instance 1 — elements: ["Book Flight (UC2)", "Validate Payment Details (UC3)"]
  Instance 2 — elements: ["Book Flight (UC2)", "Process Payment (UC4)"]
  Instance 3 — elements: ["Book Flight (UC2)", "Generate Ticket Confirmation (UC8)"]

ELEMENT FORMAT RULES
  • List exactly two elements per instance: the base use case and the inclusion use case.
  • Format: "Use Case Name (UCx)" where UCx is the PlantUML alias (e.g. UC2, UC3).
    If no alias was assigned, use just the name: "Use Case Name".
  • Do NOT include include-relationship entries (arrows) in the elements list.
    The relationship is implied by the pairing of base and inclusion.

CHAIN INCLUDES  (A →include→ B →include→ C)
  If B is itself an inclusion use case that also includes C, the chain produces two
  instances: (A, B) and (B, C). B acts as both an inclusion use case (for A) and a
  base use case (for C). List each pair as its own instance.

═══ CONSTRUCT COMPLETENESS RULE ═════════════════════════════════════════════
Before finalising the JSON, re-scan every <<include>> arrow in the PlantUML that
originates from the base use case of that instance. For each target use case:
  • If it has no direct actor association → it qualifies as a functional decomposition
    and MUST appear as a separate instance paired with the base use case.
  • If it IS directly associated with an actor → it does not qualify; exclude it.

Omitting even one qualifying inclusion use case is an error.

═══ SYSTEM CONTEXT (BOUNDARY) RULE ══════════════════════════════════════════════
Every PlantUML model MUST include a system boundary rectangle:
  • Use a `rectangle "System Name" {{ }}` block to represent the system context.
  • ALL use cases MUST be placed INSIDE the rectangle.
  • ALL actors MUST be placed OUTSIDE the rectangle.
  • The rectangle label should be the name of the system being modelled.

Example structure:
  actor Customer
  rectangle "Online Store" {{
    usecase "Browse Products" as UC1
    usecase "Place Order" as UC2
  }}
  Customer -- UC1
  Customer -- UC2

═══ ASSOCIATION RULE ═════════════════════════════════════════════════════════════
Actor-to-use-case associations MUST be undirected lines (--), NOT directed arrows (-->).
  CORRECT:  Customer -- UC1
  WRONG:    Customer --> UC1

═══ EXTEND DIRECTION RULE ════════════════════════════════════════════════════════
The <<extend>> arrow MUST point FROM the extension use case TO the base use case.
The extension use case is the optional/conditional behaviour; the base use case is
the one being extended. The base use case does NOT know about the extension.
  CORRECT:  UC_Extension ..> UC_Base : <<extend>>
  WRONG:    UC_Base ..> UC_Extension : <<extend>>
Example: "Apply Discount" extends "Process Payment", so:
  "Apply Discount" ..> "Process Payment" : <<extend>>    ✓
  "Process Payment" ..> "Apply Discount" : <<extend>>    ✗

═══ REFACTORING SCOPE RULE ═══════════════════════════════════════════════════════
